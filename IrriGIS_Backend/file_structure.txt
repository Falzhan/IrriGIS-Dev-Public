PART 4: API Endpoints Summary (IrriGIS_Backend)
================================================================================

### Authentication (/api/auth)
- POST /login                    - Email/password login
- POST /register                 - User registration
- GET /google                    - Google OAuth redirect
- GET /google/callback           - Google OAuth callback
- GET /facebook                  - Facebook OAuth redirect
- GET /facebook/callback         - Facebook OAuth callback
- GET /test                      - Protected test endpoint

### Users (/api/users)
- GET /                          - Get all users (nia_admin only)
- GET /:id                       - Get user by ID
- GET /me                        - Get current user profile
- GET /ias                       - Get irrigator associations (includes ris_id, service_area as GeoJSON)
- POST /                         - Create user (nia_admin only)
- PUT /:id                       - Update user
- PUT /:id/password              - Reset password (nia_admin only)
- DELETE /:id                    - Delete user (nia_admin only)

### Reports (/api/reports)
- GET /                          - Get all reports (with pagination, filtering, includes ticket info)
- GET /:id                       - Get single report
- POST /                         - Create report (with image upload, auto-groups to existing ticket if matching criteria)
- PUT /:id                       - Update report (with image upload, supports ticket_id for merge/unmerge)
- DELETE /:id                    - Delete report (nia_admin only)

### Ticket Settings (/api/ticket-settings)
- GET /                          - Get ticket grouping settings
- PUT /                          - Update settings (proximity_threshold_meters, auto_group_enabled)

### Tickets (/api/tickets)
- GET /                          - Get all tickets (with pagination, filtering)
- GET /:id                       - Get single ticket
- POST /                         - Create ticket manually (nia_admin only)
- PUT /:id                       - Update ticket status/assignment
- POST /:id/comments             - Add comment to ticket
- DELETE /:id                    - Delete ticket (nia_admin only)

### Ticket Sub-Statuses (/api/ticket-sub-statuses)
- GET /                          - Get all sub-statuses
- GET /:id                       - Get sub-status by ID
- GET /for-ticket?ticket_id=     - Get sub-statuses for a specific ticket
- POST /                         - Create sub-status (nia_admin only)
- PUT /:id                       - Update sub-status (nia_admin only)
- DELETE /:id                    - Delete sub-status (nia_admin only)

### Report Presets (/api/report-presets)
- GET /                          - Get all presets
- GET /:id                       - Get preset by ID
- GET /by-category?category=     - Get presets by category
- GET /categories                - Get all categories
- POST /                         - Create preset (nia_admin only)
- PUT /:id                       - Update preset (nia_admin only)
- DELETE /:id                    - Delete preset (nia_admin only)

### GIS (/api/gis)
- GET /reports                   - Get reports as GeoJSON (with spatial filtering)
- GET /features                  - Get GIS features as GeoJSON (canal lines)
- GET /features/:id              - Get single GIS feature
- POST /features                 - Create GIS feature (nia_admin only) - converts empty strings to null for UUID fields
- PUT /features/:id              - Update GIS feature (nia_admin only) - converts empty strings to null for UUID fields
- DELETE /features/:id           - Delete GIS feature (nia_admin only)
- GET /ris                       - Get river irrigation systems list (PUBLIC - no auth required)
- GET /ris/:id                   - Get RIS by ID
- GET /ias                       - Get irrigator associations list (includes ris_id, service_area as GeoJSON)
- GET /ias/geojson              - Get irrigator associations as GeoJSON
- GET /ia/:id                   - Get IA by ID
- POST /ias                     - Create IA (nia_admin only)
- PUT /ias/:id                  - Update IA (nia_admin only)
- DELETE /ias/:id                - Delete IA (nia_admin only)
- GET /stats                     - Get dashboard stats (nia_admin, nia_field_officer only)

### Notifications (/api/notifications)
- GET /                          - Get notifications (with pagination)
- GET /unread-count              - Get unread notification count
- PUT /:id/read                  - Mark notification as read
- PUT /read-all                  - Mark all notifications as read
- DELETE /:id                    - Delete notification

### Static Files
- GET /uploads/*                 - Serve uploaded report images
- GET /users/*                  - Serve user profile images

PART 5: User Roles & Permissions
================================================================================

## Access Control for IrriGIS_Admin (Web Admin Panel)

Only the following can access IrriGIS_Admin:
- nia_admin (requires is_active=true)
- ia_admin (requires is_active=true AND ia_id assigned)

Blocked from IrriGIS_Admin (use IrriGIS_App instead):
- nia_field_officer
- ia_member

## Role Permissions Matrix

| Role              | IrriGIS_Admin | IrriGIS_App | Notes                                          |
|-------------------|---------------|-------------|------------------------------------------------|
| nia_admin         | Yes           | No          | Full access, is_active=true required           |
| nia_field_officer | No            | Yes         | Use mobile app only                            |
| ia_admin          | Yes*          | No          | *is_active=true AND ia_id required            |
| ia_member         | No            | Yes         | Use mobile app only                            |

Notes:
- ia_admin registration requires admin approval (is_active=false by default)
- NIA personnel cannot register via web (must be created by nia_admin)
- Web registration (/register) is only for ia_admin role
- OAuth users: NIA emails auto-activated, others require admin activation
- ia_admin can only view/manage users within their own IA

PART 6: Key Features
================================================================================

Backend:
- JWT-based authentication with bcrypt password hashing
- Single-session enforcement (session_token in JWT + DB validation, auto-logout on new login)
- OAuth support (Google, Facebook) via Passport.js with env-based callback URLs
- Role-based access control (RBAC)
- Admin panel access restricted to nia_admin and ia_admin only (via role/is_active checks)
- ia_admin user filtering by ia_id (can only see/manage users in their IA)

 - 1 report ticket can contain multiple reports. A report is submitted, that initial report creates a ticket, and if following reports that is the same/identical will be still under the same ticket as the initial report.
 The Dual Relationship
There are actually two distinct connections happening between these tables, which is exactly what you need for an "Initial Report vs. Subsequent Reports" workflow.

   1. The "Grouping" Relationship (One-to-Many)
      The Connection: The id in report_tickets is linked to the ticket_id in the reports table.
      The Diagram Evidence: You can see the line connecting these two fields, with a "crow's foot" symbol on the reports side. This symbol signifies a "Many" relationship.
      What this means: One single report_ticket can have many reports associated with it. Whenever a new report is deemed identical to an ongoing issue, its ticket_id is just set to the existing ticket's ID, successfully grouping them together.

   2. The "Origin" Relationship (One-to-One / Reference)
      The Connection: The id in the reports table is linked to the report_id in the report_tickets table.
      What this means: This serves as the anchor. When that very first report comes in, the system creates the ticket. By saving the initial report's ID into the report_tickets.report_id column, the ticket always has a direct pointer to the original issue that spawned it, regardless of how many duplicate reports are added later.

- How the Data Flows (Step-by-Step)
   If we trace your logic through these specific tables, the lifecycle looks like this:
   Initial Submission: A citizen/user submits a report. A new row is created in reports. Let's say its id is Report-A.
   Ticket Creation: The system recognizes this is a new issue and creates a new row in report_tickets (Ticket-1).
   It sets report_tickets.report_id to Report-A (recording the origin).
   Backlinking: The system updates Report-A so its ticket_id is set to Ticket-1.
   Duplicate Submission: Later, a second report comes in for the same issue. A new row is created in reports (Report-B).
   Grouping: The system detects it's the same issue as Ticket-1. It simply sets Report-B's ticket_id to Ticket-1.

CRITICAL: Report Ticket Creation Rule (NEW):
- Only reports with category='issue' create a report_tickets entry
- Non-issue reports (inspection, maintenance, cleaning, other) are stored in reports table only
- Non-issue reports have ticket_id = NULL and display status "No Ticket" in the UI
- This prevents unnecessary tickets for routine inspection/maintenance reports
